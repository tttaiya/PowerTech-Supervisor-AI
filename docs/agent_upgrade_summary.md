# Day01-Day10 Agent 差异化改造总结

## 总体目标

把项目从基础 RAG/AIOps Demo 升级成更接近真实生产 OnCall 的智能运维助手。改造重点不是堆功能，而是增强可观测、可治理、可复盘、可评测和可演示能力。

## 每天做了什么

Day01：本地部署和环境检查。目的是保证项目能跑起来，后续优化有稳定基线。

Day02：增加 trace 主链路。目的是让每次诊断都有 trace_id 和生命周期状态。

Day03：增强节点级 trace。目的是记录 planner、executor、replanner 的中间过程。

Day04：增加工具注册表。目的是让工具有统一元数据、权限、超时和描述。

Day05：增加统一工具执行器。目的是统一记录工具成功、失败、耗时、原始输出和错误类型。

Day06：增加上下文压缩。目的是处理长日志和长工具输出，降低上下文噪声。

Day07：增加 bad case 复盘。目的是让失败样例沉淀成后续优化资产。

Day08：增加告警截图解析。目的是支持真实 OnCall 中常见的截图入口。

Day09：准备演示脚本和检查清单。目的是让项目讲解有清晰主线。

Day10：增加评测集和评测记录。目的是让优化效果可以复现、比较和回归。

## 项目亮点

- 可观测：trace 记录诊断生命周期、节点输出和工具调用。
- 稳定性：工具统一超时、错误分类和禁用开关。
- 成本控制：长输出进入上下文前先压缩。
- 复盘闭环：bad case 从失败中沉淀改进方向。
- 多入口：支持文本问题、AIOps 自动诊断和截图告警。
- 可评测：固定 eval cases 支撑回归。
