# 简历项目描述

## 简历 bullet

- 基于 FastAPI、LangChain、LangGraph 和 Milvus 搭建智能 OnCall 运维助手，支持 RAG 运维知识问答和 Plan-Execute-Replan 自动故障诊断。
- 为 AIOps 诊断链路设计 trace 可观测体系，记录 trace_id、节点执行、工具调用、重规划决策和最终报告，支持问题复盘和演示追踪。
- 设计统一工具注册表和工具执行器，支持工具元数据、启停控制、超时治理、错误分类、原始输出留存和压缩摘要。
- 实现上下文压缩服务，对长日志和长监控结果抽取服务名、时间、关键错误、异常指标和可能原因，降低 LLM 上下文噪声。
- 建设 bad case 复盘机制和评测集，覆盖 RAG、AIOps、异常、截图等场景，形成“评测发现问题、trace 定位问题、bad case 沉淀问题”的优化闭环。
- 增加告警截图解析入口，将 OCR 文本结构化为告警名称、服务、指标、当前值、阈值和时间范围，并生成诊断任务。

## 面试讲法

我这个项目不是单纯接一个大模型接口，而是围绕真实 OnCall 的工程问题做了增强：如何看清 Agent 做了什么，如何治理工具调用，如何处理长日志，如何记录失败案例，如何证明优化有效。

## 面试题

1. 你的简历里 trace 可观测具体做了什么？
2. 工具执行器和普通函数调用有什么区别？
3. 上下文压缩为什么是 AIOps 场景的关键能力？
4. bad case 机制如何体现持续优化？
5. 你如何向非技术面试官解释这个项目价值？
