# 面试讲解提纲

## 1 分钟版本

我做的是一个智能 OnCall 运维助手，支持运维知识库问答和自动故障诊断。为了让它不像普通 Demo，我重点做了工程化增强：trace 可观测、工具治理、上下文压缩、bad case 复盘、截图告警入口和评测集。这样系统不仅能回答问题，还能解释诊断过程、定位失败原因，并通过评测持续优化。

## 3 分钟版本

项目基础架构是 FastAPI 提供接口，LangChain/LangGraph 编排 Agent，Milvus 做向量检索，DashScope 作为大模型。AIOps 部分采用 Plan-Execute-RePlan：先规划排查步骤，再调用日志、监控、知识库工具，最后根据证据决定继续执行还是输出报告。

我做的差异化改造主要有三层。第一层是可观测，给每次诊断生成 trace_id，记录 planner、executor、replanner 和工具调用。第二层是稳定性，工具统一注册和执行，支持超时、错误分类、原始输出留存和摘要压缩。第三层是持续改进，bad case 记录失败案例，eval cases 做回归，截图解析扩展真实告警入口。

## 被追问时的回答方向

- 如果问为什么要 trace：因为 Agent 不是确定性程序，必须知道每一步为什么这么做。
- 如果问为什么要工具治理：因为真实工具会超时、失败、无结果或权限受限。
- 如果问为什么要上下文压缩：因为日志太长时，直接塞给 LLM 成本高且容易淹没关键信息。
- 如果问为什么要 bad case：因为 Agent 优化不能凭感觉，要把失败沉淀成可复现样例。
- 如果问未来如何做：接真实 OCR、接真实监控日志、自动化 eval、把 bad case 做向量检索。
